% ─── Abstract ─────────────────────────────────────────────────────────────────
\begin{center}
{\Large{\bf{ABSTRACT}}}
\end{center}

Financial fraud detection and regulatory sanctions screening require financial institutions
to cross-reference customer databases against third-party sanctions lists --- a process
that creates a fundamental conflict between regulatory compliance and data privacy law
(GDPR). This project addresses that conflict by designing and implementing \textbf{FLARE},
a privacy-preserving sanctions screening system
powered by post-quantum Laconic Private Set Intersection (LE-PSI).

FLARE is built on the LE-PSI protocol of D\"{o}ttling et al.\ (DKLLMR23), which
constructs laconic PSI from Ring Learning With Errors (Ring-LWE) --- a lattice-based
hardness assumption standardised by NIST (FIPS 203) and widely believed to be quantum-resistant.
The laconic property --- $O(n \log m)$ communication complexity --- is critical for
unbalanced sanctions screening where the server set $m$ (sanctions list) greatly exceeds
the client set $n$ (customer portfolio), enabling sublinear growth in server-side
communication cost.

The central engineering contribution is a batched witness generation scheme that reduces
peak auxiliary memory by 97.9\% --- from a fully-parallel worst case exceeding 300~GB
to a constant footprint compatible with commodity server hardware --- enabling the
protocol to run without specialised infrastructure. Further, a distributed
execution architecture deployed across a Google Compute Engine cluster achieves a
$15\times$ reduction in end-to-end latency, bringing execution time within the bounds
of overnight compliance batch windows. We compare empirically against ALOS22 (classical
laconic PSI baseline from pairings) and KKRT16 (OT-based PSI), and demonstrate FLARE
as a fully functional GDPR-compliant web application with a Next.js front-end, Go
back-end, and real-time screening pipeline.

\textbf{Limitations acknowledged:} The large-scale benchmark operates at 64-bit quantum security
(ring dimension $D = 256$), below the NIST-recommended 128-bit standard. Correctness at
128-bit security ($D = 2048$) is validated on small datasets. The implementation assumes
a semi-honest adversary model. These constraints and pathways to address them are
discussed in Chapter~7.
